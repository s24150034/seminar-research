5 開発方針と要件定義

5.1 なぜWebアプリケーションなのか

課題解決の手段としては、既存の時間管理システムの導入も考えられた。しかし、OCの個別相談は「当日受付」「相談分野が多数」「1日限りの短期運用」「スタッフの入れ替わりが激しい」といった独自の事情があり、汎用サービスをそのまま当てはめるのは難しかった。

そこで、次の理由からWebアプリケーションを自作する方針とした。

  • 複数端末でのリアルタイム共有：受付・各ブース担当が別々の端末から同じ状態を見られる必要があり、ブラウザさえあれば使えるWebアプリが最適だった
  • 現場に合わせた柔軟なカスタマイズ：その時の学部・部署の構成に合わせて自由に作り込める
  • 導入コストの低さ：IT課からiPabを借りる予定だったため、アプリのインストールが不要である必要があった
  • 学びの実践：データサイエンスコースで学んだPythonの知識を、実課題の解決に応用する機会になる ## 要件の整理

開発にあたり、機能要件と非機能要件を次のように整理した。

機能要件

分類 要件
ブース管理 各相談ブースの状態（空き・相談中・休憩）をリアルタイムに管理・更新できる
待機列管理 来場者を分野別に登録し、受付済みの待機者状況を一覧で把握できる
受付 新規来場者を待機列に登録できる
統計 当日の相談件数・平均相談時間を自動で集計・表示する

非機能要件

分類 要件
操作性 ITに不慣れな学生スタッフでも直感的に操作できる
応答性 状態の更新が素早く反映される
導入容易性 ブラウザのみで動作し、インストール不要
保守性 分野の追加・変更が容易な構造にする

5.2 技術選定とその理由

本アプリケーションは、以下の技術スタックで開発した。

層 採用技術 選定理由
バックエンド Python / Flask ゼミや業務で学習しており、小〜中規模のWebアプリを軽量に構築できる
O/Rマッパー SQLAlchemy Pythonオブジェクトとしてデータを扱え、DBの差し替えも容易
データベース SQLite サーバー1台で完結し、追加のDBサーバーが不要。短期運用に適する
フロントエンド HTML / CSS / JavaScript ブラウザ標準技術のみで動作し、端末を選ばない

大規模な同時アクセスが想定されないOC運営では、構成をシンプルに保てるFlask + SQLAlchemy + SQLiteの組み合わせが、開発速度・保守性・導入容易性のバランスに優れていると判断した。
